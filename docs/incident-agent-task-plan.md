# 事故调查 Agent 任务计划（证据扫荡架构版）

> 架构立场：确定性流水线 + 模型只坐关键岗位。三段式——证据扫荡（零模型）→ 分析师循环（thinking 档，有界）→ 结论过堂（强制验证）。
> 产品定位：跨系统事故调查助手，省工程师排查时间；证据为主角，结论为注脚。
> 配套文档：[incident-agent-harness-optimization.md](incident-agent-harness-optimization.md)（机制细节与 grok-build 源码依据，按章节号引用）。

## 总览

| 里程碑 | 周期 | 产出 | 上桌时刻 |
|--------|------|------|---------|
| M0 基线 | 1 天 | 回归集 + 当前表现基线 | — |
| M1 证据扫荡流水线 | 第 1 周 | 每张工单自动产出证据包 + 无结论报告 | **第 5 天可给工程师用** |
| M2 分析师循环 + 过堂 | 第 2 周 | 假设、深挖、验证、分级结论 | 第 10 天端到端 |
| M3 实测迭代 | 第 3 周起 | 省时数据 + 失败分布驱动的迭代 | 持续 |

模型调用预算（每次调查）：扫荡 0 次 / 分析循环 ≤10 轮 / 验证 1-2 次。
工具总数上限：8 个，粗粒度。

---

## M0 基线（1 天）

| # | 任务 | 说明 | 交付 |
|---|------|------|------|
| 0.1 | 建回归集 | 从真实生产工单挑 8-10 例，打三档标签：信息充分 / 有锚点 / 无锚点 | `cases/` 目录：工单原文 + 档位 + 人工认定的关键证据与根因 |
| 0.2 | 定验收口径 | 按档位定期望：信息充分→根因或强假设；有锚点→关键证据齐；无锚点→正确追问或 BLOCKED 报告。**不是每例都期望根因** | 验收表 |
| 0.3 | 记录基线 | 当前 agent 跑一遍回归集，记录每例表现 | 基线表（之后每改必对比） |

## M1 第 1 周：证据扫荡流水线（零模型调用）

原则：前十个动作是清单不是搜索——没有方向选择、没有关键词发明、不会漏查。

| # | 任务 | 估时 | 说明 |
|---|------|------|------|
| 1.1 | 锚点提取器 | 0.5d | 规则代码：从工单/告警/报错文本抽 服务名 / 时间窗 / 业务ID(user·order·trace) / 堆栈 / 报错文案；字段标注来源 filled-by-user / filled-by-tool / unknown（§3.1） |
| 1.2 | 数据源注册表 + repo 路由表 + OutOfScope | 0.5d | 可查服务枚举；服务名/包名前缀→repo 映射；注册表外目标返回类型化 `OutOfScope` + 替代查证面提示（区别于 NoData） |
| 1.3 | 证据采集器 ×6（并行扇出） | 2.5d | a. 日志检索：时间窗+级别，重复折叠、截断带 "at least N"<br>b. 堆栈抽取 + stack_jump：帧解析→路由 repo→file:line 源码 ±30 行（§4.5 原语一）<br>c. 日志骨架反查：通配变量→常量片段→定位打日志的代码（§4.5 原语二）<br>d. 部署/配置变更：时间窗内 diff<br>e. 指标异常：时间窗内可用指标的突变检测（有 metrics 源才做）<br>f. 边界检查：第三方错误形态分类（timeout/refused/4xx/5xx/SSL）+ 我方出口日志 + 横向对比（同公司其他调用方） |
| 1.4 | 证据包 schema | 0.5d | JSON：anchors / log_findings[] / stacks[] / code_locations[] / deploys[] / metrics[] / boundary / **negatives[]（"查了没有"是一等条目，排除性证据）**；每条带出处引用 id |
| 1.5 | 报告渲染 v1（无结论版） | 1d | 时间线 + 证据摆盘 + 已排除项 + 缺失项。没有结论段——本周不让模型说话 |

**M1 验收**：回归集全部案例产出证据包；有锚点档案例中，人工认定的关键证据被扫荡捕获率 ≥80%；第 5 天交给 1-2 位工程师试用真实工单，收集"这份包省了你几分钟"。

## M2 第 2 周：分析师循环 + 结论过堂

原则：模型是分析师不是采集员——第一个动作是读证据包，不是查东西。

| # | 任务 | 估时 | 说明 |
|---|------|------|------|
| 2.1 | 分析师循环骨架 | 1d | thinking 档；输入=证据包；假设板（`verification` 必填：写"从哪取证"）；有界 ≤10 轮；每轮注入 investigation-state 头（§2.3） |
| 2.2 | 定向深挖工具 ×3 | 2d | a. trace_chain：静态调用链一次返回整条线（接口→impl、MyBatis 方法→XML statement 桥接为确定性规则）<br>b. entity_mapping + 只读 SQL：JPA/MyBatis 解析出表·字段映射，SQL 从猜变填空<br>c. log_search 定向版（供扩窗/换关键词重查） |
| 2.3 | 循环护栏 | 0.5d | 无工具调用门禁（≤2 次，§2.1）+ 纪律三条进 prompt（§2.2）+ 查空分类接入（NoData/QueryError/Truncated/OutOfScope，§4.2） |
| 2.4 | 结论过堂 | 1d | 对抗验证者：refute 人设（§5.2 单员版）+ 两条机器检查（引用真实存在于本次 timeline；因果时间线自洽——因在果前）；打回 ≤1 轮 |
| 2.5 | 分级出口 + 报告 v2 | 0.5d | CONFIRMED / HYPOTHESES / ELIMINATION / BLOCKED（§6.1）；报告以证据为主角、结论为注脚；查不到明说缺什么 |

**M2 验收**：埋"伪根因"案例被验证者 refute；无锚点案例输出正确降级而非硬答；回归集端到端对比 M0 基线，逐例记录变化。

## M3 第 3 周起：实测与迭代（持续）

| # | 任务 | 说明 |
|---|------|------|
| 3.1 | 双周实测 | 1-2 位工程师真实工单使用，每单记录"省了几分钟"，攒 5-10 个数据点——这是汇报、谈判、士气的唯一硬通货 |
| 3.2 | 失败标注 | 每个失败案例标注断点位置（B1 方向 / B2 查询 / B3 工具 / B4 解读 / B5 认知 / C 出口），看分布，修占比最高的两项 |
| 3.3 | 按症状引入延后机制 | 菜单见配套文档：验证 panel（错误结论仍漏网时）/ 停滞检测（原地打转高频时）/ 压缩（调查经常 >20 轮时）/ 记忆与 dream（同类事故反复出现时）/ 子代理（多服务并行取证成为瓶颈时） |

---

## 明确不做（本计划周期内）

双模式参数化、跨会话记忆、三员 panel、gap fingerprint、doom loop、停滞分类器、上下文压缩、子代理、自动修复代码、对外宣称"自动定位根因"。

## 风险与对策

| 风险 | 对策 |
|------|------|
| 日志平台 API 限流/慢 | 扫荡结果按 (服务,时间窗) 缓存；采集器并行但限并发 |
| trace_chain 的 Java 断点覆盖不全（AOP/MQ 异步） | 断链处如实标注"链在 X 断开（异步/动态注入）"，退化为骨架反查 + 提示，不假装连续 |
| thinking 档配额/时延 | 仅分析师与验证者用；验证者可降档实测对比 |
| 扫荡证据包过大 | 每采集器输出上限 + 重复折叠；包内按证据强度排序，弱证据折叠为摘要 |

## 验收总原则

每改一刀跑回归集；每周问一次那个唯一重要的问题：**工程师拿到这份输出后，排查时间短了吗？**
